# 日内 Alpha Engine 架构升级方案 评审意见

评审日期：2026-07-24
评审对象：八层引擎架构方案（Universe / Regime / Alpha / Adaptive Threshold / Position / Execution / Pair / Online Learning）
评审基准：`src/at0/` 现有实现、`docs/strategy_optimization_implementation.md` v1.1、新 baseline（2026-07-24）

---

## 0. 总评

**方向认同，节奏反对。**

- 认同：连续评分替代布尔规则、regime 输出分值而非标签、四层回测分析（IC / 交易质量 / 分层 / 参数平台）——这些与本项目"反过拟合、可滚动验证"的既定目标完全一致，且第一阶段与现有 `evaluate_reduce/add_signal` 的三层计分结构（`extreme_score / confirm_score / layer_scores`）天然衔接，改造成本低。
- 反对：在**当前数据量（20 股 × 23 交易日 ≈ 78 笔配对交易）**上启动 Online Learning、自动权重更新、六状态分层和全面动态阈值。这不是架构问题，是统计功效问题——样本撑不起这些自由度，做出来的每一个"自适应"都是过拟合发生器。

**核心修订：把"回测分析层"从方案末尾提到最前面，把 Online Learning 从"自动更新"降级为"离线建议 + 人工审批"。**

---

## 1. 三条红线（原方案直接冲突项）

### 红线一：Online Learning 自动更新权重/阈值 违反 §9.2

`docs/strategy_optimization_implementation.md` §9.2 明确：**未经人工审核不得修改风控阈值**；§1.2 非目标明确：**不在样本不足时选择最优参数**。

原方案第八层"每天收盘自动更新 VWAP/RSI/ADX 权重，无需人工调参"与两条纪律直接冲突。且当前日均仅 3~5 笔配对交易，按日更新权重等于追逐噪声。

**修订**：Online Learning 改为 **Offline Suggestion Engine**——每周生成一份权重/阈值调整建议报告（附样本量、置信区间、分 regime 表现），人工审批后才落地，且每次落地走双跑验证。这保留了 90% 的价值，消除了 100% 的纪律冲突。

### 红线二：样本量撑不起分层与学习

- 六 regime × 行业 × 流动性分位 × 时间段的分层统计：78 笔交易摊下去，多数格子只有 0~3 笔，任何结论都是噪声。
- Alpha Vector 7+ 个权重的学习：有效样本至少需要数百到上千笔分 regime 交易。
- **P0-7（扩数据至 60+ 交易日、扩股票池）是所有高级功能的前置条件，且目前未开始。**

**修订**：数据扩容为第一优先级工程任务，早于任何新引擎开发。

### 红线三：Best-Match 配对是"统计幻觉"高危区

本项目刚踩过一次坑：止损腿盈亏漏计 `cost_reduction` 造成回测虚高。Pair Engine 需要认清一个事实：

> **在 A 股底仓做 T 场景下，真实经济盈亏 = 全部成交的现金流之和，与"哪笔买配哪笔卖"无关。配对方式只改变盈亏的**归因和展示**，不改变账户实际赚了多少钱。**

Best-Match（收益最大配对）会系统性地把"好看的配对"挑出来配平，把亏损留在未配对腿上——报表上的配对胜率和配对收益会变好看，账户实际 PnL 一分钱不变。这正是又一种统计幻觉。机构用 Best Match 主要为税务/仓位归因目的，A 股个人做 T 场景不存在这个需求。

**修订**：保留 FIFO 作为记账口径不变；把精力投到真正影响经济盈亏的地方——**每条腿的止损/持有/放弃决策**（当前 55.2% 配对失败卡在"未回归 VWAP"，这才是真问题）。回测报告可以增加"全现金流口径 PnL"作为对账基准，任何配对口径的统计必须能与它对平。

## 2. 逐层评审

| 层 | 评审结论 | 说明 |
|---|---|---|
| **Universe Engine** | ✅ 认同，缩水执行 | 硬过滤（ST/退市/停牌/次新）+ 流动性/振幅评分方向正确。但动态评分需要全市场日频数据管道，当前只有 20 股样本。先在现有 `screener.py` 上实现评分函数与 60 分门槛，全市场扫描等数据管道就绪后再上。注意：振幅门槛的经济含义应显式化——**预期日内振幅必须 > 3× 往返成本（约 0.9%）**，否则做 T 无正期望，这比"ATR<1.5% 不做"更本质。 |
| **Regime Engine** | ✅ 认同方向，⚠️ 状态数存疑 | 现有 `detect_market_regime()`（P0-6，4 状态）刚落地。升级为连续分值输出（trend_score/mean_score/vol_score）认同。但 4→6 状态在 5min 数据（一天 48 根）上，ADX/DMI 需要的窗口偏长，日内状态切换本来就少，六分类的边界样本极稀。建议：**先输出三个连续分值，状态标签保持 4 个**，用分层回测验证哪些状态区分度真实存在，再决定是否细分。 |
| **Alpha Engine** | ✅ 最认同的一层 | 布尔→连续评分是正确升级，且与现有 `layer_scores` 计分结构衔接自然。两个约束：① 初始权重**等权或手工设定**，写入 thresholds.yaml 走版本管理，不上来就学习；② 每个 score 的映射函数（如 vwap_dev→score 的折点）本身也是参数，要纳入参数平台分析，防止把过拟合从阈值搬进映射表。 |
| **Adaptive Threshold Engine** | ⚠️ 缓行 | `阈值 × VolatilityFactor × RegimeFactor` 是自由度乘法。23 天样本上全面动态化 = 过拟合机器。建议每次只动态化**一个**阈值（首选 VWAP 开仓偏离，它对 55.2% 配对失败问题最相关），用参数平台分析验证动态化后平台区是否变宽，验证通过再动下一个。止损公式 `max(1ATR, 1.5σ, OpenDev×0.8)` 属风控阈值变更，**按 §9.2 走人工审核**，且注意现有动态止损 floor 1.5% 的 Tier 1 整改本身还未完成双跑验证。 |
| **Position Engine** | ✅ 认同，注意约束 | Confidence × RiskBudget × VolAdjust 合理。补三个约束：① A 股 100 股整手取整在低价股上会显著扭曲目标仓位，要显式处理；② 底仓做 T 场景下可卖数量受底仓与 T+1 规则约束，上限不是 35% 而是 min(风险预算, 可回转额度)；③ 仓位分档函数（95→35% 等）也是参数，进版本管理。 |
| **Execution Engine** | ✅ 认同，依赖数据 | 动态限价、超时提价认同且不依赖 L2（用 1min 高低点可近似模拟成交率）。TWAP/Iceberg 对个人做 T 的单笔规模（几万~几十万）意义有限，优先级放低。 |
| **Pair Engine** | ❌ 反对 Best-Match | 见红线三。保留 FIFO，投入改为腿级止损/持有决策优化。 |
| **Online Learning** | ❌ 反对自动化，✅ 认同分析 | 见红线一。统计框架（分 regime 的 WinRate/PF/MAE/MFE/Holding）全盘保留，做成周报 + 人工审批。 |
| **回测四层分析** | ✅✅ 最高优先级 | IC/RankIC/Alpha Decay、MAE/MFE、分层回测、参数平台——这是**方案里唯一不增加任何过拟合风险、且能指导所有其他层怎么建**的部分。应该第一个做：先测量，后优化。没有 IC 分析之前，谈 Alpha 权重都是盲人摸象。 |

## 3. 修订后的实施路线

```
Phase 0（本周，硬性前置）
  ├─ Tier 1 风险整改双跑验证（遗留任务，未完成前冻结一切新开发）
  └─ P0-7 数据扩容启动：60+ 交易日 × 尽量宽的股票池

Phase 1（1~2 周）：测量层（原方案的"回测系统设计"提前）
  ├─ backtest.py 增加 IC / RankIC / Alpha Decay 分析
  ├─ 交易质量指标：MAE / MFE / Expectancy / Holding Time
  ├─ 分层回测（先按现有 4 regime + 时间段两个维度）
  └─ 参数平台扫描工具（先扫 VWAP threshold 一个参数）
  产出：哪些现有信号真的有预测力 → 决定 Alpha Engine 建什么

Phase 2（2~3 周）：连续评分改造
  ├─ evaluate_reduce/add_signal 输出 AlphaScore（0~100），等权起步
  ├─ detect_market_regime 增加 trend/mean/vol 三个连续分值
  ├─ 双跑验证：连续评分 vs 现有布尔门槛，同数据对比
  └─ Universe 评分函数落地（现有 20 股上先跑通）

Phase 3（2~4 周）：单点动态化 + 仓位
  ├─ 仅动态化 VWAP 开仓阈值（× vol factor），平台分析验证
  ├─ PositionEngine：Confidence 分档仓位（含整手/可回转额度约束）
  └─ 动态止盈止损方案 → §9.2 人工审核 → 双跑 → 落地

Phase 4（数据就绪后评估）：
  ├─ Offline Suggestion Engine（周报 + 人工审批，替代 Online Learning）
  ├─ regime 是否 4→6：由 Phase 1 分层结果决定
  └─ L2 订单流：先回答"数据从哪来、什么成本"再立项
```

与原方案的关键差异：
1. **测量先于优化**——四层回测分析从最后提到最前。
2. **双跑验证贯穿每阶段**——延续既定纪律，每个新引擎与旧实现同数据对比后才切换。
3. **删除 Best-Match 配对，Online Learning 降级为人工审批制。**
4. **L2 依赖项单列**，不进主线排期。

## 4. 决策建议

| 决策项 | 建议 |
|---|---|
| 整体方向（规则策略 → 评分制 Alpha 平台） | **采纳** |
| 原四阶段路线 | **不采纳**，按上文修订路线执行 |
| 当前一步 | **先完成 Tier 1 双跑验证 + 启动数据扩容**，两者未完成前不动 strategy.py |

一句话：这套架构的天花板取决于数据和验证纪律，不取决于引擎数量。八个引擎里最值钱的是"回测四层分析"这个不起眼的部分——先把尺子造准，再造武器。
